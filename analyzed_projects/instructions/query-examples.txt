# Query Examples for ScriptBee Data Model

These examples demonstrate how to query the in-memory graph via the `execute_code` MCP tool.
`graph_data` is a pre-initialized `MCPSandboxView` over the typed v2 Graph — use it directly.

NOTE on metric names: trait names like `anomaly.testing.BugMagnet`, overview names
like `authorship`, and relation kinds like `cochange` are shown as concrete strings
here for clarity, but the canonical list lives in code. Call `list_metrics()` once
per session to see the current catalog — new metrics show up there automatically.
If a name in this file no longer appears in `list_metrics()`, the metric was renamed;
treat the example as a pattern template.

---

## 1. Project overview: name and commit count

```python
num_commits = len(graph_data.commits.all())
# `project_id` is on the typed Graph (and on the view via __getattr__).
print(f"Project '{graph_data.project_id}' has {num_commits} commits.")
```

## 2. Top 5 most modified files

```python
from collections import Counter

# `graph.changes` is the typed ChangeRegistry — it has a `by_file` index
# keyed on `change.file_ref` (an EntityRef). The view proxies through.
file_counter = Counter()

for change in graph_data.changes:
    file_counter[change.file_ref.id] += 1

top_5_files = file_counter.most_common(5)
print("The 5 most modified files (with number of modifications):")
for fname, count in top_5_files:
    print(f"  {fname}: {count} modifications")
```

## 3. Which user contributed to the most Jira issues?

```python
from collections import defaultdict

# Walk commits, look up issue keys mentioned in each message, and tally
# distinct issues per author.
user_issue_count = defaultdict(set)

for commit in graph_data.commits.all():
    for issue in commit_issues(commit, graph_data):
        user_issue_count[commit.author_ref.id].add(issue.key)

if user_issue_count:
    author_id, issues = max(
        user_issue_count.items(), key=lambda kv: len(kv[1])
    )
    print(f"Author {author_id} contributed to {len(issues)} Jira issues (the most).")
else:
    print("No user found with associated issues.")
```

## 4. Top 5 "bug magnet" files (cross-project query)

```python
from collections import Counter

# Walk every bug-typed issue, follow its linked commits via the inverse
# of commit_issues (issue_commits), and count file touches per commit.
bug_file_counter = Counter()

bug_type = next(
    (t for t in graph_data.issue_types.all() if t.name.lower() == "bug"),
    None,
)
if bug_type is not None:
    bug_type_id = bug_type.id
    for issue in graph_data.issues.all():
        if issue.type_ref.id != bug_type_id:
            continue
        for commit in issue_commits(issue, graph_data):
            # graph.changes.by_commit is the ChangeRegistry's per-commit index.
            for change in graph_data.changes.by_commit[commit.ref()]:
                bug_file_counter[change.file_ref.id] += 1

print("Top 5 bug magnet files:")
for fname, count in bug_file_counter.most_common(5):
    print(f"  {fname}: {count} occurrences in bug-related commits")
```

## 5. Pull requests with linked JIRA issues

```python
# The `pr_issue` relation links PRs to issues; iterate over the index.
for pr in graph_data.pull_requests.all():
    rels = graph_data.relations.for_source(pr.ref())
    issue_keys = [r.target.id for r in rels if r.relation_kind == "pr_issue"]
    if issue_keys:
        print(f"PR #{pr.number} '{pr.title[:50]}' -> Issues: {', '.join(issue_keys)}")
```

## 7. Co-change neighbours of a file

```python
# `cochange_neighbors` returns File entities. To get strengths too,
# read graph.relations directly:
from src.common.kernel import EntityKind, EntityRef
src_ref = EntityRef(kind=EntityKind.FILE, id="src/server.py")
edges = [
    r for r in graph_data.relations.of_kind_in_window("cochange", "lifetime")
    if r.source == src_ref
]
edges.sort(key=lambda r: -r.strength)
for r in edges[:10]:
    print(f"{r.strength:>5.0f}  {r.target.id}")
```

## 9. Filter classifiers (e.g. weekend bugfixes)

```python
from src.common.kernel import EntityKind

# Classifiers live in graph_data.classifiers; iterate by-target and look
# for commits whose (nature, weekday) pair matches the predicate.
weekend_bugfixes = []
for commit in graph_data.commits.all():
    by_dim = graph_data.classifiers.for_target(commit.ref())
    nature = by_dim.get("message.nature")
    weekday = by_dim.get("weekday")
    if (
        nature is not None and nature.value == "bugfix"
        and weekday is not None and weekday.value in ("sat", "sun")
    ):
        weekend_bugfixes.append(commit)
print(f"{len(weekend_bugfixes)} weekend bugfix commits")
```

## 9b. Files over 1000 LOC (Lizard B1)

```python
# graph_data.file_metrics is the typed FileMetricRegistry. In v2 each
# FileMetric is one (file_ref, metric_name, value) row — the legacy
# per-metric scalar fields (sum_nloc / max_ccn / avg_ccn / function_count
# / longest_function_nloc) collapsed into a single value column keyed
# by metric_name. To see all metrics for one file, iterate and group
# by m.file_ref.
big = sorted(
    (m for m in graph_data.file_metrics
     if m.metric_name == "sum_nloc" and m.value >= 1000),
    key=lambda m: -m.value,
)
for m in big[:10]:
    print(f"{int(m.value):>6} NLOC  {m.file_ref.id}")
```

## 9i. Top files by Insider code-smell count (B4)

```python
# graph_data.quality_issues is the typed QualityIssueRegistry. In v2
# QualityIssue carries `file_ref: EntityRef` (was `file_path: str` on

qi = graph_data.quality_issues
if len(qi) == 0:
    print("No Insider data ingested.")
else:
    from collections import defaultdict
    per_file = defaultdict(int)
    for issue in qi:
        per_file[issue.file_ref.id] += issue.occurrence_count
    for fid, total in sorted(per_file.items(), key=lambda kv: -kv[1])[:10]:

```

## 9l. Files carrying a given App Inspector concern (B5)

```python
# graph_data.app_tags is the typed AppTagRegistry. One AppTag entity per
# (file_path, tag) pair, ingested from a `<repo>-chronos-tags.json` upload.
# Each entry carries `project_ref`, `file_ref`, `file_path`, `tag` (full
# dotted taxonomy starting with `appinspector.`, e.g.
# `appinspector.OS.Network.Connection.Socket`), and `strength: int`.
#
# AppTagRegistry indexes (all multi=True):
#   by_file[file_ref]       -> tags on a given file
#   by_project[project_ref] -> all tags in a project
#   by_tag[tag]             -> entities with exactly this tag string
#   by_tag_root[root]       -> entities whose tag starts with
#                              `appinspector.<root>.` (e.g. "OS",
#                              "Cryptography")
ats = graph_data.app_tags
if len(ats) == 0:
    print("No App Inspector data ingested.")
else:
    # All files touching the OS / network surface, ranked by strength.
    net = sorted(
        ats.by_tag_root["OS"],
        key=lambda t: -t.strength,
    )
    for t in net[:10]:
        print(f"  {t.strength:>3}  {t.tag:<55}  {t.file_path}")
```

## 10. Commits per author (top 10)

```python
from collections import Counter

author_counts = Counter()
for commit in graph_data.commits.all():
    # Resolve the author EntityRef to the typed GitAccount entity.
    account = graph_data.git_accounts.get(commit.author_ref.id)
    if account is not None:
        author_counts[account.name] += 1

print("Top 10 contributors by commit count:")
for name, count in author_counts.most_common(10):
    print(f"  {name}: {count} commits")
```

## Usage Notes

- `graph_data` is pre-initialized (an `MCPSandboxView` over the typed v2 Graph). Do not re-declare it.
- `graph_data` exposes registries directly: `graph_data.commits`, `.issues`, `.pull_requests`, `.files`,
  plus enrichment registries (`.traits`, `.classifiers`, `.relations`, `.components`) and the rest
  of the typed Graph fields via read-through `__getattr__`.
- Cross-entity navigation: use `commit_issues(commit, graph_data)`, `pr_commits(pr, graph_data)`,
  `issue_commits(issue, graph_data)` (free helpers) — the old `commit.issues` property is gone.
- Do NOT mutate the graph data — it is shared in-memory across executions.
- Use `print()` for all output. The result is captured stdout.
- Keep output concise: summarize, aggregate, or limit to top N.
- Standard library modules (collections, datetime, etc.) are available.
- Access registries via `.all()` for a snapshot tuple, `.get(id)` for lookup, or iterate directly.
